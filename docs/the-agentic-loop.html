A field guide in three parts

The Agentic Loop

Build a persistent, multi-agent development system you fully control: GitHub issues in, reviewed pull requests out, running 24/7 on a VPS, steerable from your phone — with you making exactly three kinds of decisions and agents doing everything else.

Part 1 is pure concepts — every piece of the system, with diagrams, before you type a single command. Part 2 is the build: by its end you have the complete system installed and idling — orchestrator, agents, skills, hooks, budgets, Telegram, VPS — wrapped around a blank repo. Part 3 puts your hands on the wheel: build a real app end-to-end, add a feature, fix a bug. Steps are numbered; under each one, a How it works box explains what the code does and ties it back to Part 1. Terracotta Note boxes flag design decisions, amber Careful boxes flag risks, green Done boxes mark checkpoints.

Part 1 · The big pictureWhat you're building

The loop in one sentence: an orchestrator watches GitHub for issues labeled agent:ready, moves each one through a state machine (design → design review → human gate → code → code review → human gate → merge), and every model call is routed so you can dial cost and capability per role.

Terracotta gates are the only places you act. Everything between touch points 1 and 3 — the “hands-off middle” — is touch point 2: you watch, you don't type.

Three properties define the whole design, and every section of this guide traces back to one of them:

  • State lives in GitHub, not in any process. An issue's label is its state. Kill the orchestrator any time; restart it; nothing is lost. And you can steer from anything that can flip a label — a terminal, the GitHub phone app, a Telegram tap.
  • You steer at exactly three points. Shape the brief and the design (highest leverage), stay hands-off through the middle, validate and merge at the end. Everything else is the loop's job.
  • Nothing certifies its own work. The coder's claim of "green" is re-checked in a sandbox it can't influence, then by CI, then by reviewers from other model families, then by you. Layered, mutually distrustful verification is what makes autonomy safe.

Why each piece earns its place:

  • GitHub as the database. The naive design keeps state in the orchestrator's memory — then a crash loses everything. Putting state in labels means the loop can die and restart mid-issue with zero loss, you can inspect everything from the GitHub mobile app, and the issue timeline doubles as a complete audit log of what every agent did.
  • A dumb orchestrator, smart workers. The Python daemon makes no decisions about content — it only routes: which issue, which stage, which model, when to stop. Intelligence lives in the model calls; control lives in code you can read in five minutes. When something behaves weirdly, you always know which of the two to debug.
  • Roles instead of one super-agent. Splitting the work into narrow roles gets you three things: cheaper routing (a review is a smaller job than writing code, so it gets a cheaper model), sharper prompts (each role has one job and an output contract), and independent judgment — the reviewer never saw the coder's reasoning, so it judges the artifact, not the intention.
  • Files as memory. Model calls are stateless — every call starts from nothing. All continuity comes from files the orchestrator injects: AGENTS.md (durable conventions), the per-issue memo (what's happened on this issue so far), skills (on-demand procedures). Files are debuggable: when an agent acts oddly, you can open the exact text it saw.
  • Worktrees as workbenches. Each issue gets a private checkout of the repo. Two issues in flight can't overwrite each other, and a misbehaving coder can't touch your main checkout.

Three deliberate asymmetries hide in this timeline. Human effort is front-loaded — steps 3–6 (design) get most of your attention because errors there are cheapest to fix; by step 10 you're verifying, not designing. And every automated step ends by handing evidence to a skeptic — coder → Docker re-check → CI → cross-family reviewers → you. Nothing certifies its own work.

Part 1 · The big pictureThe three loops of iteration

"Agentic loop" is really three loops nested inside each other, each with its own cadence, its own verifier, and its own failure response. Understanding which loop you're in at any moment is most of understanding the system:

Loop 3 · The steering loop — youcadence: hours–days

Cycle: brief → design rounds → approve → (wait) → validate → merge, or bounce back with a repro. Verifier: your judgment, running the real app. On failure: a precise comment + a label flip — never a hand-edit.

Loop 2 · The outer loop — CI on every PRcadence: minutes

Cycle: PR opened/updated → clean VM → install → lint → test → green check or red X; release on merge. Verifier: a fresh environment that doesn't trust anyone's machine. On failure: the PR is blocked; the failure feeds back into the coder's next attempt.

Loop 1 · The inner loop — the codercadence: seconds

Verifier: the test suite + linter, run locally in the worktree. On failure: read the error, fix, rerun — dozens of times if needed, at zero cost to you.

edit code → make lint && make test → read failure → fix → ↺ … until green

Three things to internalize about this picture:

  • Speed lives inside; trust lives outside. The inner loop is fast and runs on the coder's own word; the outer loop is slower but runs on a machine nobody configured by hand; your loop is slowest and carries the authority. Each layer catches what the layer inside missed — the coder's tests can be wrong, CI can't check that the feature makes sense, only you can do the last part.
  • The coder must live inside loop 1. A coder that can see a failing test and fix it in seconds beats one that gets a single blind attempt per orchestrator pass. This is why the coder runs on a real harness (Claude Code) rather than a bare completion call — the quality of the whole system tracks the tightness of its tightest feedback loop.
  • Bounce-backs are loop 3 doing its job. When you return a PR with a repro instead of fixing it, you're feeding the failure into loop 1's next cycle and into the memo — so the fix, and the lesson, both land inside the system instead of in your working memory.

Part 1 · The big pictureMemory & skills — how agents know things

Model calls are stateless, so everything an agent "knows" beyond its training is text you hand it at call time. The art is in when each kind of knowledge loads. The system uses four layers:

LayerLoadsScopeHolds
AGENTS.mdAlways, at session startWhole repo, every toolDurable conventions: commands, structure, style, security rules, do-not-touch list
CLAUDE.mdAlways (Claude Code)Whole repoA thin @AGENTS.md shim + Claude-only notes
SkillsOn demand, when relevantTask-scopedProcedural know-how: a checklist, a framework's conventions, a validation sequence
Memos / scratchpadsInjected per issueWorking setThe spec, reviews, decisions, error logs for one issue

The discipline that makes this work: always-loaded things stay tiny; everything conditional becomes a skill. A bloated always-on file is a tax paid on every call by every agent — in tokens, in dollars, and in attention (models follow instructions worse as instruction volume grows). A skill splits the cost: its one-line description is always visible (~30 tokens); its full body (~500+ tokens) loads only when a task matches. That split is called progressive disclosure, and it's why you can install twenty skills and pay for none of them until the moment one is needed.

A skill is just a folder with a SKILL.md — YAML frontmatter (name, description, allowed tools), then markdown instructions. Here's the shape (you'll install five real ones in Part 2.7):

markdown---
name: oauth-security-checklist
description: Use when reviewing any code or design that implements OAuth login,
  session handling, or Google sign-in.        ← the always-loaded trigger
allowed-tools: Read, Grep                     ← least privilege: look, don't touch
---

# OAuth security review checklist             ← loads only when auth is in scope
1. State/CSRF — the authorization request sends a `state` param and verifies it…
2. Secrets — client id/secret from env vars, never literals, never committed…
…

Two more facts worth knowing. Skills are an open standard — the same file works in Claude Code, Codex, Gemini CLI, Cursor — so nothing you write is locked to one harness. And why markdown-files-as-memory at all, rather than some vector database? Because files are debuggable: when an agent behaves oddly, you open the exact text it saw. Boring, inspectable, versionable memory wins at this scale.

AGENTS.md is the cross-tool conventions standard (read by Codex, Cursor, Copilot, Gemini CLI, Aider…); Claude Code reads CLAUDE.md, so you keep one source of truth and a one-line shim (@AGENTS.md). What belongs in it: the stack, exact build/test/lint commands, structure, security rules, a do-not-touch list. What doesn't: anything a formatter enforces, full API docs, "write clean code" filler, anything conditional — that's a skill.

Part 1 · The big pictureAgents, reviewers & trust

What a role agent is

Nothing exotic: a system prompt plus a model choice. The system prompt is a job description with three parts — identity ("You are the FEASIBILITY design reviewer"), focus (the specific questions to ask), and an output contract ("Output ## Concerns and ## Verdict"). The contract matters more than it looks: downstream code and future-you both parse these outputs, so a predictable shape makes reviews comparable across issues. Personality words ("terse and skeptical") do real work too — they push the model away from its default agreeable register, which is exactly wrong for a reviewer.

The full identity of an agent in this system is the sum of six things:

agent = system prompt      (its role and voice)
      + model              (via its own budget-capped key)
      + AGENTS.md          (always-on shared memory)
      + skills             (on-demand expertise, tool-restricted)
      + tools/connectors   (least privilege, enforced by hooks & allowlists)
      + memo               (what it has learned on this issue)

Why reviewers come in pairs, from different families

Diversity beats collusion. If reviewer and coder are the same model — or two reviewers share a family — they share blind spots and tend to bless each other's mistakes ("evaluator collusion"). Reviewers from different model families catch different classes of bug, which is the cheap mitigation. The same logic gives reviewers fresh context: they see the artifact, never the coder's reasoning, so they judge what was built, not what was intended.

How trust is enforced, not requested

Instructions like "never touch .env" are requests to a language model — usually honored, never guaranteed. The system therefore layers mechanical guardrails, each independent of the others:

GuardrailMechanismWhat it stops
Human gatesIssues park at agent:needs-human; only a human label-flip proceedsAnything merging without you
Tool allowlistsThe coder may run make, pytest, git commit — not curl, not pip install, not git pushEscapes and surprises on your machine
HooksA separate OS process vets every file edit; exit code 2 blocks itEdits to .env, keys, migrations — regardless of what the model decides
SandboxesValidation re-runs in a Docker container with no network, seeing only the worktreeHallucinated dependencies phoning home; host contamination
Budget capsOne spend-capped API key per role, enforced server-side; plus an in-loop per-pass capA runaway loop burning your account
Least-privilege accessReviewers get read-only everything; only the coder writes; write-scoped tokens, not one god-tokenA reviewer "helpfully" rewriting the code it judges
One step per passThe loop advances each issue a single state per passA bug spiraling through the pipeline in seconds

Notice the pattern: every guardrail lives outside the model — in the harness, the OS, the provider, or GitHub. That's the rule to carry forward: enforcement beats instruction.

The philosophy is the durable part: cheap models on frequent roles, spend on the coder, reviewers from different families.

Money is controlled in four independent layers, from "free" to "nice to have": your account balance (a hard ceiling — keep it low while learning), per-role capped keys (the main lever — enforced by the provider, so no bug of yours can overspend), the in-loop cap (the orchestrator stops itself per pass), and a local cost dashboard (visibility: spend by role and by day). A realistic solo loop lands in low single-digit dollars per day; the caps exist for the bad day — the runaway retry — not the normal one. You'll set all four up in Part 2.9.

A label is just a string attached to an issue — but treat it with discipline (exactly one agent:* stage label per issue at a time, and moving the label is the state transition) and GitHub becomes your workflow database, for free. Whenever the loop needs you, the issue lands in agent:needs-human and nothing proceeds until you move the label. That's your steering wheel — and it works identically from a terminal, the GitHub app on your phone, or the Telegram bot you'll build in 2.10.

Step 4 — commit the outer loop

Loop 2 from Part 1: CI that validates every PR in a clean environment and releases on merge. It goes in before any agent works, so the very first implementation PR gets gated:

bashmkdir -p .github/workflows
cat > .github/workflows/ci.yml <<'EOF'
name: CI
on:
  pull_request:
  push:
    branches: [main]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.11" }
      - run: pip install -e . && pip install pytest ruff
      - run: ruff check .
      - run: pytest -q          # OAuth is mocked in tests — CI never calls Google
  release:
    needs: validate
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: echo "Deploying 🎉 (placeholder for fly deploy / docker push)"
EOF

cat > .gitignore <<'EOF'
.env
.venv/
worktrees/
orchestrator/logs/
role_keys.json
pending.json
__pycache__/
EOF

git add AGENTS.md CLAUDE.md .github .gitignore
git commit -m "Agentic scaffolding: conventions, labels, CI"
git push

on: declares the triggers: every pull request, and every push to main (which is what a merge is). Each job runs on a brand-new Ubuntu VM — that freshness is the entire value of CI: nothing from your laptop, the agent's worktree, or the last run leaks in, so "works in CI" means reproducibly works. The validate steps mirror the inner loop deliberately — checkout, install Python, install the project, lint, test — same checks, cleaner room. The release job shows two workflow-control keywords worth knowing: needs: validate (don't even start unless validate passed) and the if: expression (only on a push to main — i.e., only after a merge, never on a mere PR). Replace its echo with a real deploy command when you have somewhere to ship to. Until Lab 1 builds the app, CI will fail on a fresh PR — expected: there's no pyproject.toml yet. The gate exists before the thing it gates.

Cost tracking, per-role keys, and budget data all live here and only here."""

from openai import OpenAI
from dotenv import load_dotenv

# --- per-role keys (provisioned in 2.9; falls back to the single key until then) ---
_keyfile = pathlib.Path(__file__).parent / "role_keys.json"
ROLE_KEYS = json.loads(_keyfile.read_text()) if _keyfile.exists() else {}
_FALLBACK = os.environ["OPENROUTER_API_KEY"]
_clients: dict[str, OpenAI] = {}

def _client_for(role: str) -> OpenAI:
    if role not in _clients:
        _clients[role] = OpenAI(base_url="https://openrouter.ai/api/v1",
                                api_key=ROLE_KEYS.get(role, _FALLBACK))
    return _clients[role]

# --- cost accounting: in-memory tally for the loop's cap + CSV log for the dashboard ---
_usd_spent = 0.0
_LOG = pathlib.Path(__file__).parent / "logs" / "costs.csv"

def _log_cost(role, model, cost, usage):
    _LOG.parent.mkdir(exist_ok=True)
    new = not _LOG.exists()
    with _LOG.open("a", newline="") as f:
        w = csv.writer(f)
        if new:
            w.writerow(["ts", "role", "model", "cost_usd",
                        "prompt_tokens", "completion_tokens"])
        w.writerow([datetime.datetime.now(datetime.UTC).isoformat(), role, model,
                    f"{cost:.6f}",
                    getattr(usage, "prompt_tokens", ""),
                    getattr(usage, "completion_tokens", "")])

def add_cost(role: str, model: str, cost: float):
    """For spend that doesn't flow through call() — e.g. the headless coder."""
    global _usd_spent
    _usd_spent += cost
    _log_cost(role, model, cost, None)

def call(role: str, model: str, system: str, user: str, max_tokens: int = 4000) -> str:
    """One LLM turn: system prompt + user message in, text out. Tracks spend
    via the cost OpenRouter reports on the response itself."""
    global _usd_spent
    resp = _client_for(role).chat.completions.create(
        model=model,
        messages=[{"role": "system", "content": system},
                  {"role": "user", "content": user}],
        max_tokens=max_tokens,
        extra_body={"usage": {"include": True}},   # ask OR to return actual cost
    )
    usage = getattr(resp, "usage", None)
    cost = float(getattr(usage, "cost", 0) or 0) if usage else 0.0
    _usd_spent += cost
    _log_cost(role, model, cost, usage)
    return resp.choices[0].message.content

def spent() -> float:
    return _usd_spent

  • The chokepoint pattern. Every completion call goes through call(), and every call names its role first. Most people know one checkout per clone. A worktree is git's built-in way to have several: git worktree add -b agent/issue-7 worktrees/issue-7 main creates a new folder that is a full, real checkout — starting from main, on a brand-new branch — while sharing the original repo's .git object database. That sharing is why it's near-instant and nearly free (no second copy of history), and why a branch committed in the worktree is immediately visible to the main checkout and pushable from either. For this loop it means: issue #7's coder works in worktrees/issue-7, issue #8's in worktrees/issue-8, neither can see or clobber the other's uncommitted files, and your own checkout stays pristine. One rule to remember: git refuses to check out the same branch in two worktrees — which here is a feature, since each issue owns exactly one branch.

Part 2 · Set it all up2.6 The loop & the coder

The heart. One pass advances every ready issue by one state, so a crash never loses more than a step and a bug can't spiral through the pipeline in seconds.

The file is long but has only four moving parts — recognize them and the whole thing is legible:

  • The memo (_issue_memo / _append): an append-only markdown notebook per issue — the scratchpad layer from Part 1. Every stage reads it before acting and appends its output after. When the coder "knows" what the design reviewers said, it's because those reviews sit in this file and get pasted into its prompt.
  • Stage handlers (run_design_draft, run_design_reviewers, …): one function per state. Each follows the same rhythm — gather context → call a role → record the output (memo + issue comment) → move the label. The label move at the end IS the state transition.
  • The STAGES table: the entire state machine in six lines, label → handler. Want a new stage? Add a label and a function and one row here. This table is the answer to "where do I customize the workflow."
  • one_pass(): for each stage, for each issue wearing that label — budget check, run the handler, catch and report errors. Then main() just repeats passes forever with a nap.

Two properties fall out of this structure. Restartability: no state lives in the process, so kill it any time; the next pass re-reads reality from GitHub and picks up exactly where things stood. Debuggability: any weird behavior traces to either a handler (deterministic Python — read it) or a model call (open the memo and see exactly what the model was given and what it said).

Step 1 — the state machine

Create orchestrator/loop.py:

python"""The agentic loop. One pass = advance every ready issue by one state.
State is tracked entirely in GitHub labels, so the loop is restartable."""
import os, re, json, subprocess, time, pathlib
from rich.console import Console
from dotenv import load_dotenv

from llm import call, spent, add_cost
from agents.roles import ROLES
import gh, worktree

load_dotenv()
con = Console()
MEM = pathlib.Path(__file__).parent / "memory"
MAX_USD = float(os.environ.get("MAX_USD_PER_RUN", "2.0"))

# ---------- the per-issue memo (the scratchpad memory layer) ----------
def _issue_memo(n: int) -> pathlib.Path:
    p = MEM / f"issue-{n}.md"
    p.parent.mkdir(exist_ok=True)
    if not p.exists():
        p.write_text(f"# Issue {n} scratchpad\n\n")
    return p

def _append(path: pathlib.Path, section: str, text: str):
    with path.open("a") as f:
        f.write(f"\n## {section}\n\n{text}\n")

# ---------- your feedback, gathered from wherever you left it ----------
def _human_feedback(issue, branch: str) -> str:
    """Touch point 1 happens on the design PR, so read comments from the
    issue AND the open PR — conversation tab and inline review comments."""
    fb = [c.body for c in gh.repo.get_issue(issue.number).get_comments()
          if not c.body.startswith(("###", "⚠️"))]          # skip the bot's own
    pr = gh.open_pr_for_branch(branch)
    if pr:
        fb += [c.body for c in pr.get_issue_comments()]      # conversation tab
        fb += [f"[on {c.path}] {c.body}" for c in pr.get_review_comments()]
    return "\n".join(fb)

# ---------- stage handlers: gather → call role → record → move label ----------
def run_design_draft(issue):
    """Draft or revise DESIGN.md on a branch; open/update a design PR.
    You iterate via PR comments; approving the PR is the sign-off."""
    memo = _issue_memo(issue.number)
    wt = worktree.create(issue.number)
    branch = f"agent/issue-{issue.number}"
    existing = (wt / "DESIGN.md").read_text() if (wt / "DESIGN.md").exists() else "(none yet)"
    feedback = _human_feedback(issue, branch) or "(none yet)"

    prompt = (f"App brief (from the issue):\n{issue.title}\n{issue.body}\n\n"
              f"Existing design (may be empty):\n{existing}\n\n"
              f"Human feedback to incorporate this round:\n{feedback}\n\n"
              f"Output ONLY the full updated DESIGN.md content.")
    design = call("design", ROLES["design"]["model"], ROLES["design"]["system"],
                  prompt, max_tokens=6000)

    (wt / "DESIGN.md").write_text(design)
    subprocess.run(["git", "add", "DESIGN.md"], cwd=wt, check=True)
    subprocess.run(["git", "commit", "-m", f"Design for #{issue.number}"],
                   cwd=wt, check=False)                      # no-op if unchanged
    subprocess.run(["git", "push", "-u", "origin", branch], cwd=wt, check=False)

    pr = gh.open_pr_for_branch(branch) or gh.open_pr(
        branch, f"Design: {issue.title}",
        f"Design doc for #{issue.number}. Comment to request changes; "
        f"**approve this PR to sign off** and start implementation.")
    _append(memo, "Design draft", f"See {pr.html_url}")
    gh.comment(issue.number,
               f"### 📐 Design draft ready → {pr.html_url}\n\n"
               f"**Touch point 1:** comment on the PR + relabel `agent:design-draft` "
               f"to redraft, or approve the PR + relabel `agent:design-approved`.")
    gh.set_label(issue.number, "agent:needs-human",
                 ["agent:design-draft", "agent:ready"])

def run_design_reviewers(issue):
    """Two design reviewers (different families) critique the approved design.
    You arbitrate, then relabel agent:coding to implement."""
    memo = _issue_memo(issue.number)
    wt = worktree.create(issue.number)
    design = (wt / "DESIGN.md").read_text()

    feas = call("review_feasibility", ROLES["review_feasibility"]["model"],
                ROLES["review_feasibility"]["system"], design)
    sec = call("review_security", ROLES["review_security"]["model"],
               "You are the SECURITY design reviewer. Focus on the OAuth flow, "
               "secret handling, session security, and data exposure. "
               "Output ## Concerns and ## Verdict.", design)

    _append(memo, "Design review — feasibility", feas)
    _append(memo, "Design review — security", sec)
    gh.comment(issue.number,
               f"### 🔍 Design reviews\n\n**Feasibility**\n{feas}\n\n"
               f"**Security**\n{sec}\n\n---\n**Touch point 1 (arbitrate):** "
               f"relabel `agent:coding` to implement, or `agent:design-draft` to revise.")
    gh.set_label(issue.number, "agent:needs-human", ["agent:design-approved"])

def run_code_review(issue):
    memo = _issue_memo(issue.number)
    a = call("review_correctness", ROLES["review_correctness"]["model"],
             ROLES["review_correctness"]["system"], memo.read_text())
    b = call("review_security", ROLES["review_security"]["model"],
             ROLES["review_security"]["system"], memo.read_text())
    _append(memo, "Review A (correctness)", a)
    _append(memo, "Review B (security)", b)
    gh.comment(issue.number,
               f"### ✅ Code review\n\n**Correctness**\n{a}\n\n**Security**\n{b}")
    gh.set_label(issue.number, "agent:needs-human", ["agent:code-review"])

# ---------- the state machine, in one table ----------
STAGES = [
    ("agent:ready",           run_design_draft),      # fresh issue → first draft
    ("agent:design-draft",    run_design_draft),      # you asked for a revision
    ("agent:design-approved", run_design_reviewers),  # you approved the design PR

    ("agent:trivial",         run_code),              # fast path: skip design
    ("agent:code-review",     run_code_review),
]

def one_pass():
    for label, handler in STAGES:
        for issue in gh.issues_with_label(label):
            if spent() >= MAX_USD:
                con.print(f"[red]Budget cap ${MAX_USD} hit — stopping.[/]")
                return

            except Exception as e:

                gh.comment(issue.number, f"⚠️ Agent error: `{e}`")

        one_pass()
        con.print(f"[dim]pass complete — spent ${spent():.3f} — sleeping 30s[/]")
        time.sleep(30)

    main()

Step 2 — the coder stage

The coder is the one role that must live inside loop 1 (Part 1): it needs to run tests, read failures, and fix them — so it runs on a real harness. Add this to loop.py (above the STAGES table, which already references run_code):

python# ---------- the coder: Claude Code headless inside the worktree ----------
ALLOWED = ("Read,Glob,Grep,Edit,Write,"
           "Bash(make:*),Bash(pytest:*),Bash(ruff:*),"
           "Bash(git add:*),Bash(git commit:*)")

def _coder_headless(wt: pathlib.Path, memo_text: str) -> tuple[bool, str, float]:
    """Run Claude Code as the coder. Returns (ok, summary, cost_usd)."""
    prompt = (
        "You are implementing an approved design in this repo. Read AGENTS.md "
        "and DESIGN.md first. Implement the spec below, working test-first. "
        "Run `make lint` and `make test` and iterate until BOTH are green. "
        "Then `git add -A` and `git commit` with a clear message. Do NOT push.\n\n"
        f"--- Spec, design notes and reviews ---\n{memo_text}")
    p = subprocess.run(
        ["claude", "-p", prompt,
         "--permission-mode", "acceptEdits",
         "--allowedTools", ALLOWED,
         "--output-format", "json",
         "--max-turns", "100"],
        cwd=wt, capture_output=True, text=True, timeout=3600)
    try:
        out = json.loads(p.stdout)
        return (not out.get("is_error", True),
                out.get("result", "")[:2000],
                float(out.get("total_cost_usd", 0)))
    except (json.JSONDecodeError, KeyError):
        return False, (p.stdout + p.stderr)[-2000:], 0.0

def _inner_loop_in_docker(wt: pathlib.Path) -> tuple[bool, str]:
    """Independent re-check in a throwaway container. The agent's own 'it's
    green' claim is never the thing that opens the PR."""
    cmd = ["docker", "run", "--rm", "--network", "none",
           "-v", f"{wt}:/app", "-w", "/app", "python:3.11-slim",
           "bash", "-lc",
           "pip install -q -e . 2>/dev/null; pip install -q pytest ruff; "
           "ruff check . && pytest -q"]
    p = subprocess.run(cmd, capture_output=True, text=True, timeout=300)
    return p.returncode == 0, (p.stdout + p.stderr)[-3000:]

def run_code(issue):

    wt = worktree.create(issue.number)
    branch = f"agent/issue-{issue.number}"

    ok, summary, cost = _coder_headless(wt, memo.read_text())
    add_cost("coder", "claude-code", cost)               # same books as every role
    _append(memo, "Coder session", f"ok={ok} cost=${cost:.4f}\n\n{summary}")

    green, log = _inner_loop_in_docker(wt)
    _append(memo, "Independent inner loop", f"passed={green}\n\n```\n{log}\n```")
    if not (ok and green):
        gh.comment(issue.number,
                   f"### 👩‍💻 Coder pass — ❌ not green yet\n```\n{log}\n```\n"
                   f"Staying in `agent:coding` for retry next pass.")
        return                            # next pass's coder starts from this error

    subprocess.run(["git", "push", "-u", "origin", branch], cwd=wt, check=True)
    pr = gh.open_pr(branch, f"Issue #{issue.number}: {issue.title}",
                    f"Closes #{issue.number}\n\nCoder inner loop green; "
                    f"independently re-validated in Docker.")
    gh.comment(issue.number, f"### ✅ Inner loop green → opened {pr.html_url}")
    gh.set_label(issue.number, "agent:code-review",

  • -p <prompt> ("print" mode) = headless: instead of an interactive chat, Claude Code takes the prompt, plans, edits files, runs commands, iterates — a full agentic session — then prints its final result and exits. Same engine you use interactively, no human in the middle.
  • cwd=wt is quietly the most important line: the session starts inside the issue's worktree, so "the repo" from the agent's point of view is that isolated checkout — its file reads, edits, test runs, and commits all land there and nowhere else. It also picks up your CLAUDE.md → AGENTS.md and skills automatically, because they're in the checkout.
  • --permission-mode acceptEdits: file edits are auto-approved (no human to click "allow"), but everything else still needs to clear the allowlist.
  • --allowedTools grammar: a bare name (Read, Edit) permits that tool; Bash(make:*) permits only shell commands starting with make. So this coder can run make test but not curl, not pip install, not git push — the least-privilege row from Part 1's guardrail table, made real.
  • --output-format json turns the final output into machine-readable fields: is_error, result (its closing summary), total_cost_usd (actual metered spend — logged via add_cost into the same books as every other role), num_turns.
  • --max-turns 100 is the runaway brake: a session that hasn't converged in 100 tool-use turns gets cut off rather than looping forever on a doomed approach.

docker run --rm --network none -v {wt}:/app -w /app python:3.11-slim bash -lc "…" — five flags, five guarantees. --rm: the container is destroyed after the run; nothing persists. --network none: no network interface at all — code under test can't call home, fetch a malicious package, or leak anything. -v {wt}:/app: only the worktree is mounted into the container; the rest of your disk doesn't exist from inside. -w /app: start there. python:3.11-slim: a fresh, minimal, pinned environment — matching CI far better than your laptop does. pythondef _write_files_from_output(wt: pathlib.Path, output: str):
    blocks = re.findall(r"# path:\s*(.+?)\n```[a-zA-Z]*\n(.*?)```", output, re.S)

        dest.write_text(content)
        written.append(rel.strip())

        if any(t in issue_text.lower() for t in triggers):

Here's the exact sequence when the coder tries to edit a file. (1) The harness sees a pending Edit call and checks the matcher — a regex over tool names; Edit|Write matches both edit paths. (2) It runs your command as a normal shell process, piping the pending call to its stdin as JSON: {"tool_name": "Edit", "tool_input": {"file_path": "...", …}}. (3) Your script reads that JSON, applies whatever logic you like — here, "is the path one of the protected ones?" — and exits. (4) The harness interprets the exit code: 0 = allow the call to proceed; 2 = block it, and whatever the script printed to stderr is shown to the agent as the reason, so it can adapt instead of retrying blindly. The security property comes from step 2 being a separate OS process: the model never sees the hook, can't negotiate with it, and can't "decide" to skip it — enforcement beats instruction, the rule from Part 1's guardrail table. def _poll_forever():
    offset = None
    while True:
        try:
            params = {"timeout": 30}
            if offset: params["offset"] = offset
            r = httpx.get(f"{API}/getUpdates", params=params, timeout=40)
            for upd in r.json().get("result", []):
                offset = upd["update_id"] + 1
                if "message" in upd:
                    _handle_message(upd["message"])
                elif "callback_query" in upd:
                    _handle_callback(upd["callback_query"])
        except Exception as e:
            print("bot error:", e); time.sleep(5)

    if not ENABLED:
        return
    threading.Thread(target=_poll_forever, daemon=True).start()
    _send("🤖 Agent loop bot online. `/status` to see what's cooking.")
  • Long polling (_poll_forever): getUpdates?timeout=30 is a normal HTTP GET that Telegram's server deliberately holds open for up to 30 seconds, answering the instant something arrives. You get push-like latency with zero infrastructure — no public URL, no open ports, no webhook endpoint to secure. The offset = update_id + 1 line is the acknowledgment: it tells Telegram "processed everything up to here," so nothing is delivered twice (and nothing is lost if the bot restarts — unacknowledged updates are re-sent).
  • The pending-gate mailbox (pending.json): the loop and the bot are two threads that need to share "which issues are parked, and what should an approval do." Rather than in-memory state, they share a tiny JSON file: the loop writes "issue 12 waits at a design gate; approval applies label X", the bot reads it, notifies you, and flips notified so you're pinged exactly once. A file rather than a queue, so a restart loses nothing — the same durability philosophy as labels-as-state.
  • Callback buttons (_handle_callback): each inline button carries a callback_data string like approve:12. A tap sends a callback_query update; the bot splits it into action + issue number and performs the same label flip you'd do in a terminal. answerCallbackQuery stops the button's loading spinner; editMessageText rewrites the original message so stale buttons can't be tapped twice.
  • The daemon thread (start): daemon=True means the bot thread dies automatically when the loop process exits — no orphan process. Both threads share gh.py, so every phone action is literally the same operation as its terminal equivalent, with the same audit trail on the issue.

## Observed
A brand-new user who signs in for the very first time sees `login count: 0` on `/me`.

## Expected
The first successful sign-in should count as 1. Nth sign-in shows N.

## Repro
1. Use a Google account that has never logged into this app before (or delete its row from the DB).
2. Sign in once.
3. Visit /me → it shows "login count: 0". It should show 1.

## Notes
Likely an off-by-one in where login_count is incremented in the OAuth callback. Add a regression test: first login yields count 1, second login yields count 2. Keep the OAuth provider mocked in tests.
'

Observed / Expected / Repro, plus an explicit ask for a regression test with exact values (1, then 2). The repro tells the coder how to reproduce; the exact values let reviewer A verify the fix is real rather than "tests are green." Note the label: agent:trivial instead of agent:ready — the STAGES table routes it straight to the coder, skipping the design phase, because a full design round for a one-line fix is waste. Judging which path an issue deserves is a loop-3 steering decision you'll make constantly.

3.3 Watch, then verify the red→green arc

  • The coder should move the increment so it runs on every successful login and add the two-login regression test — which fails before the fix and passes after. That before/after is loop 1's whole epistemology: a regression test that never failed proves nothing.
  • Reviewer A must confirm the test asserts count 1 after the first login specifically. If it rubber-stamps a vaguer test: bounce, same move as Lab 2.

3.4 Confirm locally, ship

bashgh pr checkout <IMPL_PR> && make install
make run    # fresh sign-in → /me shows 1; sign out/in → 2
make test && make lint
gh pr review <IMPL_PR> --approve --body "First login counts as 1; regression test covers it."
gh pr merge <IMPL_PR> --squash --delete-branch
gh issue edit <ISSUE> --add-label agent:done --remove-label agent:needs-human

Triage → fix → regression test → ship, with your local run and CI as the final gates — and the design phase correctly skipped.

Part 3 · The labsWhat you've actually learned

  • Design is where steering has the highest leverage. Bugs caught in the design PR cost a comment; after implementation, a full re-run. That's why touch point 1 is the long one.
  • Three touch points, only three. Steer the design, stay hands-off through the middle, do the final review. Resisting the urge to jump in during touch point 2 is the discipline that lets one person run many loops.
  • Your job is briefs and gates, not typing. A sharp brief, good design pushback, an honest final review — including running the app for real, which no agent can do for you.
  • Verification is layered, and nothing certifies its own work. The coder's inner loop, the orchestrator's independent Docker check, CI, two cross-family reviewers, your hands. Each catches what the previous missed.
  • Correct the system, not the code. A rubber-stamped missing test means: comment (it feeds the memo), or add a line to AGENTS.md, or upgrade that reviewer's model — one line in roles.py. The loop improves by editing its parts, not by you doing its job.

@claude on GitHub — a zero-daemon side channel

Tier 3 from Part 1: anthropics/claude-code-action lets you comment @claude fix the login-count bug on any issue from the GitHub mobile app, and a cloud runner implements it and opens a PR. It bypasses your design gates and reviewers — so treat it as the agent:trivial path for when you're away from everything, not a replacement for the loop:

yaml# .github/workflows/claude.yml
name: Claude
on:
  issue_comment: { types: [created] }
  issues:        { types: [opened] }
jobs:
  claude:
    if: contains(github.event.comment.body || github.event.issue.body, '@claude')
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write
      issues: write
      id-token: write
    steps:
      - uses: actions/checkout@v4
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}

The on: triggers fire whenever an issue is opened or commented on; the if: guard then checks the text actually mentions @claude, so ordinary comments don't burn runner minutes. The permissions: block sets the scopes of the run's short-lived GITHUB_TOKEN — write access to contents, PRs, and issues for minutes, not forever: least privilege at the credential, the same rule as everywhere else in this guide. Your API key rides in as a repository secret. Check the action's current README for exact inputs — like model slugs, they drift.

Scaling patterns

  • Parallelism. Worktrees already isolate issues — run several through the loop at once. Add a merge-conflict check before opening each PR.
  • Scheduled maintenance agents. A cron entry (or a scheduled cloud agent) that files agent:ready issues on a timer: dependency bumps, CI health, stale-PR triage. Same loop, a timer instead of you filing the brief.
  • A watchdog. A tiny separate process watching costs.csv and issue activity; it kills the loop (or pages you via bot.notify) if spend spikes or an issue ping-pongs between states more than N times. Cheap insurance for 24/7 running.
  • A Discord front-end — if you want a channel per role and to watch the agents like a team chat. Same mapping as the Telegram bot (lock to your user ID, loop vocabulary only, honest merges), with discord.py and one channel ID per role.
  • Upgrade rituals. When a role keeps underperforming: sharpen its skill, then its prompt, then its model — in that order of cost. The costs.csv by-role view tells you what an upgrade actually costs you per day.

The loop is the product. Keep the leash exactly as long as your verifiers are strong — and when in doubt, add a verifier, not autonomy.
